## 2026-06-27 — Test layout: default `uv run pytest` scoped to the core suite
- Decision: set `testpaths = ["tests"]` in the root `[tool.pytest.ini_options]` so the bare `uv run pytest` collects only the root package's 75-test suite; the `web/` trainer's tests run explicitly via `uv run --package by-heart-web pytest web/tests`. README's "run the tests" wording updated to match.
- Why: a fresh clone's `uv sync` installs only the root `by-heart` project, not the `web/` workspace member, so `by_heart_web` is absent from the root interpreter. Default collection walked `web/tests/`, whose module-level `from by_heart_web import viz` then raised `ModuleNotFoundError`, aborting the entire session (exit 2, zero tests run) — the first thing a judge hits after `uv sync && uv run pytest`. Scoping `testpaths` makes the failure structurally unreachable: pytest never scans `web/tests` on the bare command, independent of which workspace members happen to be installed.
- Alternatives considered: make `by_heart_web` importable at the root env (document `uv sync --all-packages`, or add `by-heart-web` to the root dev group) so one command covers web too — rejected: coupling the trimmed root project to the web/GCP deps contradicts the deliberate root/web split (blueprint §9), and the web suite already has a first-class per-package command. Guarding the web test's import with `importorskip` — rejected: it silently skips real web coverage on a full env and still misreports "the same checks run under `uv run pytest`."
- Concept served: Deployability / Documentation.
- Shows in: code, writeup.

## 2026-06-26 — Web trainer viz: activity-aware graph collapse + no-cache static serving
- Decision: make both live-graph panels collapsible, and **auto-collapse Graph A (Build Pipeline) the moment a recall session starts** — re-expanding it on Build/Re-plan — so the **Reasoning Log** grows into the reclaimed height (default 150→200px, up to 42vh while Graph A is collapsed). A green "✓ complete" badge stays on Graph A's collapsed title bar; both graph titles are real `<button>`s with a caret (keyboard-focusable), so the collapse is also a manual override. Separately, serve the front-end with `Cache-Control: no-cache` on both `/` and `/static/*`.
- Why (UX): Graph A runs once at build and is then static for the whole recall session (only Re-plan re-runs it), while the Reasoning Log is busiest precisely during recall — so the least-live element was occupying ~140px directly above the most-live one. Tying the collapse to *session start* (not to the build) keeps the full provenance→MCP→curriculum_plan pipeline visible while it actually executes, then hands its space to the log once it goes idle. The viewer's eye stays on whatever is live (Graph B + the log) — the point of the panel — without losing the proof that the build pipeline succeeded.
- Why (no-cache): surfaced by a real failure during this change — the running server was serving the new HTML/JS (verified by curl *and* by the server's own access log showing the browser's build/course/memory calls landing on it), yet the browser kept rendering the *old* page. API calls (POST/GET) are never cached, so the app *functioned*; but the cached static document/JS drew stale UI against a live backend, and Graph A never collapsed. `FileResponse`/`StaticFiles` set ETag/Last-Modified but no `Cache-Control`, so a reload could reuse the in-memory copy without revalidating. `no-cache` forces revalidation on every load (keeping the cheap ETag 304 fast-path) so a stale page can never again render against a live backend — a concrete demo-recording hazard removed.
- Alternatives considered: **manual toggle only** (closest to the original ask) — rejected as the *primary* behavior because it relies on the user remembering to collapse and the log can't grow until they do; kept as the manual override. **Grow the log only, leave the graphs alone** — rejected because the two graphs still push the log down, so the page scrolls on a laptop instead of reclaiming in-view height. **Version-stamp query on `app.js`** (`app.js?v=N`) instead of headers — rejected: it needs manual bumping and doesn't fix the already-cached *document*; `no-cache` fixes both with no upkeep. **`no-store`** — unnecessary; `no-cache` already guarantees freshness while preserving 304 revalidation.
- Concept served: Deployability (the live web-trainer agent showcase).
- Shows in: code (`web/`), video (the build pipeline lights up, then yields the stage to the live recall loop + reasoning log).

## 2026-06-25 — Web trainer recall UX (additive): retry-on-miss as fresh Graph B runs, escalating scaffold ladder, give-up reveal, solved-word persistence
- Decision: complete the recall interaction in the web trainer so a real learner can actually drill — three behaviors, all driven by the *unchanged* `recall_session` graph (`app/` still untouched, the change is confined to `web/`): **(1) retry-on-miss** — a missed word stays open and can be re-attempted; **(2) an escalating scaffold ladder across retries** — each retry climbs the §3 cue-withdrawal levels (1 rhyme cue → 2 first letter → 3 meaning gloss) instead of restarting at level 1; **(3) a "reveal & skip" give-up** — the learner can disclose the held-back word and move on. Plus a UI fix: words the learner has **earned (green) or revealed (amber) stay filled in** as the stanza re-renders for the next blank (the prior build wiped them on every new target).
- Why each is realized the way it is:
  - **Retry = a fresh Graph B run per attempt, NOT a back-edge in the graph.** The §4 loop is already realized *across* `RequestInput` resume cycles (one attempt = one `start → pause → submit → memory_update` arc), so a retry is simply another `POST /api/recall/start` for the same `(stanza,line,word)` target. This needs zero graph change, reuses the exact two-request pause/resume the web layer already had, and — a real bonus — each retry records another attempt in Learner Memory, so a struggling word organically strengthens the adaptive re-plan signal. The rejected alternative — adding a `scaffold → present_masked_line` back-edge so the loop lives inside one ADK session — would have edited the frozen `recall_session` graph and broken the "`memory_update` is terminal; the loop is across invocations" invariant the graph's own docstring states.
  - **Escalation is achieved by seeding session state, not by changing the coach.** Each retry is a new ADK session, so its `scaffold` node would normally see `prior_hint_level = 0` and hand out a level-1 hint forever. The web client tracks the strongest hint reached for the current word and passes it as `prior_hint_level` to `start_recall`, which seeds it into the new session's `state["hint_level"]`; the *existing* `scaffold` node already reads that as its `prior` and its `_validate_hint` floor (`min(3, prior+1)`) makes the ladder climb. So the graduated-cue-withdrawal pedagogy the blueprint describes becomes visible to a live user with no node edit. Verified live: ladder went 1 ("rhymes with 'know.'") → 2 ("starts with 't.'") → learner got it.
  - **The answer is disclosed only on an explicit give-up.** The target word is never sent to the browser until earned (the established §8 policy). Reveal makes "I give up" an explicit, separate request (`POST /api/recall/reveal`) rather than ever shipping the word with a grade. To support it, grading now clears only the *consumed pause* (`clear_pause()` nulls the ADK `session_id`/`interrupt_id` so a spent `RequestInput` can't be double-submitted) but **keeps `target_context`**, so a miss remains retriable and revealable; the next `start_recall` overwrites the context with the next word. A revealed word is recorded client-side as `earned:false` and rendered in a distinct dashed-amber style, so "earned green" stays meaningful.
- Scope check: this is incremental UX completion of the post-MVP trainer (still beyond the blueprint's deferred-front-end line), not new pedagogy — the grading, crutch tagging, scaffold authoring, and provenance all remain in `app/`. Kept the trunk safe: `app/` untouched, key-free smoke extended (reveal discloses only on request + the pause/context lifecycle; blanks now carry their position), full suite green, ruff clean. Direct user-reported drivers: "the correct word should remain visible after the user has entered it" and "it does not seem possible to try, try again (or … skip forward)."
- Alternatives considered: **graph back-edge for the retry loop** — rejected (edits the frozen graph, breaks the terminal-`memory_update` invariant; the across-invocations loop already models retries for free). **Re-deriving the hint level on the server from recorded attempts** — rejected as more coupling for no gain; the client already knows the level it last saw, and passing it through is a one-field, stateless hop. **Shipping the answer with the miss grade and letting the client decide when to show it** — rejected; it puts the secret in the browser before it's earned, defeating the held-back-answer policy. **Auto-revealing after level 3** — rejected; the learner should choose to give up (reveal is an explicit button), and forcing a reveal removes the "keep trying" agency.
- Concept served: ADK (the human-in-the-loop `RequestInput` loop + the scaffold node, now fully drivable live) + the crutch-removal pedagogy; Security (answer stays server-side, disclosed only on explicit request; the spent pause can't be replayed).
- Shows in: code (`web/`), video (a learner missing, watching the Scaffolding Coach climb the ladder, then succeeding — a stronger live agent beat than a single correct recall), writeup (this entry).

## 2026-06-25 — Web trainer (post-MVP, additive): FastAPI + live graph visualization, GCP-portable
- Decision: add a **new, purely additive** human-facing web app under a `web/` package that lets a real person memorize the Frost poem while **watching the agents execute live**, driving the *existing* ADK graphs (`build_pipeline`, `recall_session`) and Prosody MCP **unchanged** — `app/` is imported by reference, never edited or forked. The web layer is a thin async driver over the same `Runner` + `RequestInput` pause/resume mechanics `app/demo.py` established; it adds no pedagogy. Three sub-decisions, resolved with the user: (a) **build the course on demand, live** in the browser (so Graph A's provenance → MCP → curriculum_plan lights up node-by-node, the strongest showcase of the agents) rather than pre-building at startup; (b) package it as a **separate `web/` service with its own `pyproject.toml`** joined as a **uv workspace member** (path-depends on the real `by-heart` package), not as a root optional-dependency group; (c) **declare `google-cloud-aiplatform`** in `web/pyproject.toml` as the brief requested, even though nothing in the MVP imports it.
- Why this is in scope at all: the blueprint (§9) explicitly *defers* a custom web front-end, so this is an explicit, user-directed extension **after** the freeze-ready MVP — not a rubric requirement. It is built so it cannot destabilize the frozen MVP: `main` stays green, `app/` is untouched, the only root-repo change is one `[tool.uv.workspace]` stanza + the mechanical `uv.lock` regen. As a bonus it banks the optional **Deployability** 5th concept (§5) and gives the video a far stronger live ADK/MCP showcase than the CLI runner.
- Real-time visualization (the load-bearing piece) — grounded in two ADK facts confirmed by probing the installed package, not assumed: **(1) static topology** is read straight from `Workflow.graph.nodes`/`.edges` (each edge carries `route` ∈ {None, admit, refuse, advance, scaffold}), so the page draws exactly the graph the runtime executes — no hand-maintained duplicate that could drift. **(2) live signal** comes from an official `BasePlugin` registered via `Runner(plugins=[...])`; its `on_event_callback` fires per event before persistence. A probe corrected the design here: the clean active-node name is `event.node_info.name` (a computed property — `'provenance_gate'`, `'present_masked_line'`, …), **not** `event.author` (which is the *workflow* name, `'build_pipeline'`); the routed branch is `event.actions.route`; the human-in-the-loop pause is `bool(event.long_running_tool_ids)`. The plugin pushes a compact transition onto a per-web-session `asyncio.Queue`; `before_tool_callback` surfaces Prosody MCP calls — **filtered to the three real tools** (`pronounce`/`scan_line`/`analyze_poem`) because ADK's internal structured-output plumbing (`set_model_response`) also arrives via that hook and would misrepresent the MCP. Transport is **SSE** (`text/event-stream`, server→client only, Cloud-Run-friendly, no WebSocket upgrade/idle caveats), with heartbeats; the page draws both graphs as **hand-rolled inline SVG, no CDN** (both are tiny/fixed — 5 and 6 nodes), toggling CSS classes by node name.
- Implementation notes (so a future reader isn't surprised): a **recall spans two HTTP requests** because the §4 loop is realized across `RequestInput` resume cycles — `POST /api/recall/start` drives Graph B to the `present_masked_line` pause and stashes the paused ADK `session_id` + `interrupt_id` + presented `target_context` on the web session; `POST /api/recall/submit` resumes via `FunctionResponse(id=interrupt_id, name="adk_request_input", response={"recall": clean})`. The **same plugin instance (same queue)** is attached to every `Runner` the web session spins up, so a single open SSE stream sees the whole build→recall arc. The typed recall is **sanitized at the web boundary** (`sanitize_recall`) before it enters the FunctionResponse and reaches any model — the graph sanitizes again internally (defense in depth); the **target word never leaves the server** (only the underscore-blanked stanza/line and the word length do). The §13.4 single-loop hazard is satisfied for free by FastAPI's one persistent event loop (the "Event loop is closed" race came from multiple `asyncio.run()` calls, not one-loop concurrency); a process-wide `InMemorySessionService` singleton is shared, with per-browser isolation via distinct ADK `session_id`s and an opaque cookie `learner_id` (minimal-PII preserved).
- GCP portability: Vertex is **env-only** — `app/models.py` already loads `.env` and anticipates `GOOGLE_GENAI_USE_VERTEXAI`; the web service reaches Vertex by importing the same `gemini()` factory, so `GOOGLE_GENAI_USE_VERTEXAI=true` + `GOOGLE_CLOUD_PROJECT` + `GOOGLE_CLOUD_LOCATION` flips it with **no app code change**. `google-cloud-aiplatform` is declared (per the brief) but imported by nothing — `google-genai` already routes to Vertex — so it is isolated to `web/pyproject.toml`, kept out of the deliberately-trimmed root tree; it `uv lock`s cleanly (107 packages, no conflict). A `web/Dockerfile` (uvicorn `0.0.0.0:$PORT`, no `.env` in the image) makes it Cloud-Run-deployable.
- Verified end-to-end against the running server (real Gemini): Graph A built a 4-session Frost course with genuine Deletion Rationales in ~22s, streaming `provenance_gate route=admit → prosody_analysis` (with **`Prosody MCP · analyze_poem`**) `→ deletion_rationale → curriculum_plan`; Graph B presented `His house is in the village ______;`, paused (waiting), and on a wrong recall streamed `adjudicate route=scaffold → scaffold → memory_update`, grading `miss` with the scaffold hint *"It rhymes with 'know.'"* (level 1) and the crutch tag correctly `none`. Key-free smoke (`web/tests/test_web_smoke.py`: topology equals the real graphs, the transition envelope, the SSE generator, the provenance refusal) is green; the full suite is 80 passed + 5 web + the 5 live tests skipped key-free; ruff clean on `web/`.
- Alternatives considered: **root optional-dependency group** for the web/GCP deps — rejected because it injects fastapi/uvicorn/aiplatform into the curated root manifest (trimmed of GCP deps on purpose, §9) and muddies the always-green core; the workspace member keeps them in the deployable unit only. **WebSocket** transport — rejected; the flow is strictly server→client and SSE avoids the upgrade/idle-timeout complexity on Cloud Run. **A CDN graph library (cytoscape/vis-network)** — rejected; the graphs are too small to justify a runtime script-tag dependency, and inline SVG keeps the demo fully offline/auditable. **Pre-building the course at startup** — rejected because driving Graph A live is itself the showcase. **Forking/vendoring `app/`** into a standalone repo — rejected outright; the whole point is to drive the *real* graphs so they cannot drift.
- Concept served: Deployability (the optional 5th concept) + a stronger live demonstration of the ADK multi-agent graphs and the Prosody MCP.
- Shows in: code (`web/`), video (the trainer + the live graph highlight is the new money shot), writeup (this entry + `web/README.md` are its raw material).

## 2026-06-25 — Documentation + demo (§13 #8): one-command runner + README + DEMO_SCRIPT
- Decision: ship build item 8 as (a) a thin one-command demo runner `app/demo.py` (`uv run python -m app.demo`) **and** documented `agents-cli` playground steps — both demo paths, not one. The runner drives the five DoD §11 steps end-to-end (provenance refusal → build + Deletion Rationale → fed recall with crutch tag → adaptive re-plan → security eval). (b) Fill the README's missing practical half: a problem→solution lead on the NYT contrast, the agent/node/tool/services taxonomy (§4 "state this in the README"), embedded two-graph diagrams, a Setup section, and a Run-the-demo section. (c) Add `PUBLIC_DOMAIN.md` (the §7/§8 provenance summary the README links — it did not exist). (d) Add `DEMO_SCRIPT.md`, a ≤5:00 shot list mapped to the §7 video beats with verified commands.
- Why: the README was missing everything a judge follows (setup, run-the-demo, taxonomy, provenance link) — the §7 Documentation rubric is 20 near-free points. And there was **no single end-to-end demo entry point**: the only way to drive the graphs was the playground with hand-pasted JSON, and the recall session both pauses on `RequestInput` and depends on `ctx.state["poem_text"]` being seeded — awkward to reproduce (DoD §11 "clone and follow the README") and to record. A one-command runner that seeds the recall state and feeds each `RequestInput` pause makes the whole adaptive loop reproducible and the video friction-free to re-take. Verified: `uv run python -m app.demo --reset` prints all five steps and exits 0 with a key; the full suite is 80 green; ruff clean.
- Money-shot design (driven by testing — two real constraints discovered, not free choices): the demo's recall step is scripted to make DoD §11 #4 ("the next session strips a *different* crutch") **visibly** true and reproducible on every take. (1) **A rhyme-leaner can't show a reorder.** The default policy already strips rhyme first (rung 1) and the adaptive directive can only pull a crutch *earlier*, so "pull rhyme earlier" is a no-op on ordering. And a probe showed the live Adjudicator won't attribute a correctly-known content word to "rhythm" (tags `crutch_dependence=none`), so a *relied-on* meter signal isn't producible. The fix uses the **miss pattern** (blueprint §4 step 4 allows this): the learner *near-misses* the words the meter carries; meter dependence is then read deterministically from the word's `crutch_class` via `missed_at` (not an LLM judgment), so it is stable. The re-plan reliably pulls metrical-regularity removal from session 2 to session 1 — a genuinely reordered schedule the runner prints as a before/after table. (2) **Anchor switched from Dickinson to Frost for a reliable per-attempt crutch tag.** Testing the Dickinson build showed the per-attempt `rhyme_partner` tag was a coin-flip: Dickinson's rhymes are *slant* (`me`/`Immortality`), so the grader honestly can't decide "did they lean on it?" and tags `none` ~half the time — fine behavior, bad for a guaranteed-good recording. A probe found no Dickinson word reliably yields a non-`none` tag. Frost's strong end-rhymes (`know`/`though`/`snow`) tag `rhyme_partner` 3/3 in probes, so the demo now uses `frost-stopping-by-woods`: the learner aces `though` (reliable `rhyme_partner` tag — the crutch-tag beat), then near-misses `house`→`barn`, `horse`→`donkey`, `lake`→`river` (the §7 near-miss beat + the meter pattern that drives the reorder). The injection/PII eval still uses a Dickinson stanza internally — independent of the demo anchor. Exact masked words are pinned by `(stanza, line, word)` index; reproduced identically across runs.
- Resolved by asking the user (the one blueprint-deferred choice): §13's open-decision-points explicitly defers the demo-UI choice ("the agents-cli local playground is the low-effort default; say if you'd prefer a thin custom front-end"). Presented playground-only vs. a thin runner vs. both, with pros/cons grounded in §7/§9/§11. The user chose **Both** (and a `DEMO_SCRIPT.md` for video prep). The blueprint defers exactly this, so asking was the right call rather than self-resolving.
- The runner reuses the **single-shared-event-loop discipline** (§13.4): steps 1–4 run on one `asyncio.run`, and step 5 runs the eval's **deterministic core** (key-free, no second loop) rather than the live sweep — a second `asyncio.run` for the live eval would reuse the module-level agents' genai client across loops and trip ADK's "Event loop is closed" teardown race (the same hazard that kept the live eval sweep out of pytest). The full live injection sweep stays the standalone `evals` command, featured as its own shot in `DEMO_SCRIPT.md`. The runner is **key-graceful**: steps 1 and 5 need no key, so a clean clone runs the spine and exits 0 without a Gemini key; with a key it runs the whole loop. A key-free smoke test (`tests/integration/test_demo_runner.py`) keeps `main` green and bisectable; the live build/recall/re-plan stays out of the gate, exercised by hand.
- Implementation notes (so a future reader isn't surprised): the gate input is a **bare poem id** in the message text (passing `{"poem_id": ...}` JSON makes `_as_poem_id` treat the whole string as the id and refuse it); the `RequestInput` resume is a `FunctionResponse` part whose `id` matches the interrupt id ADK emits in `long_running_tool_ids` (name `adk_request_input`); the harness reads the word the graph chose to mask from the presented `target_context` and feeds it back (simulating a learner who recalls correctly and so leans on the still-visible cue); and the graded result is read from the **Learner Memory store** (the authoritative §13.5 record), not re-parsed from the event stream.
- Alternatives considered: **playground-only** (blueprint default, zero new code) — rejected by the user for reproducibility/record-ability, though it remains documented as the visual path; **a thin runner only** — rejected in favor of also documenting the playground (which reinforces the agents-cli lifecycle concept and is the richer visual); running step 5's **live** eval inside the demo loop — viable on the one shared loop but unnecessary and slower, so the demo runs the deterministic core and points to the standalone live command; **an image diagram** for the README — used embedded ASCII instead (no binary asset, diffable, renders in any viewer).
- Concept served: Documentation (the 20-pt rubric item) + the video deliverable (§7); reinforces the Security artifact (DoD #5) and the provenance posture by making both runnable in one command.
- Shows in: code (`app/demo.py`, the README, `PUBLIC_DOMAIN.md`), video (the runner *is* the demo spine; `DEMO_SCRIPT.md` is its shot list), writeup (README + this log are its raw material).

## 2026-06-24 — Security (§13.7): STRIDE skill + injection/PII eval + recall-input validation
- Decision: bank the Security concept by hardening the one untrusted entry point, proving the guarantees with a runnable eval, and authoring a STRIDE `SKILL.md`. (a) **Recall-input validation** lives in a small pure module `app/security/recall_input.py` (`sanitize_recall`) applied at the single choke point `_recall_text` — it bounds length, strips Unicode control/format codepoints (zero-width, bidi, BOM, soft hyphen) and collapses newlines, while preserving legitimate poetic tokens (`o'er`, `café`, hyphens). One change covers both the adjudicator read and the stored `last_recall` the coach reuses. (b) **Prompt hardening**: both LLM instructions now state the recall is untrusted DATA, never an instruction (the coach also: never reveal the word). (c) The **structural containment was already present** — `_validate_adjudication` clamps `outcome` to the legal vocabulary and `crutch_dependence` to the cues that were actually visible, and `_validate_hint` clamps the hint level — so this entry promotes the §8 validated-proposal pattern to a stated security property: a fully-injection-swayed model still cannot emit an out-of-vocabulary grade, fabricate a crutch tag, or over-escalate a hint. (d) **Minimal-PII is structural** — `Attempt` has no free-text recall field, so a recall's PII/injection payload is never persisted; no code change, the eval asserts it. (e) A new `.claude/skills/stride-threat-model/SKILL.md` maps each STRIDE category to By Heart's concrete control and its code location (a third authored Skill).
- Why: §5/§8 require Security as one of the four banked concepts, anchored in the MVP. The system's attack surface is small by design (no accounts, no open-ended ingestion, one untrusted input), so the work is to make the existing controls explicit, hardened, and provable rather than to add machinery. DoD §11 #5 ("run the injection/PII eval and see it pass") is the gradeable artifact. Verified: the eval's 28 checks pass and the standalone harness exits 0 with the live model resisting every injection scenario (each injected recall graded a non-success: `miss`/`near_miss`).
- Eval shipped as a **runnable `evals/` harness + a deterministic pytest core**, NOT an agents-cli eval-set. §8/§10 name agents-cli for the lifecycle, but DoD #5 only requires the eval *runs and passes on a clean clone*, and the settled project convention (§13.5/§13.6) is a key-free deterministic core plus a key-gated live layer. Security *properties* (injection resistance, PII non-leakage) are property assertions, not reference-answer matches, so agents-cli's eval-set format is an ill fit and key-dependent. `evals/scenarios.py` holds the synthetic injection + PII corpus as readable data; `evals/injection_pii_eval.py` runs it as a one-command PASS/FAIL report (`uv run python -m evals.injection_pii_eval`) and is imported by `tests/eval/` so `uv run pytest` stays the single gate.
- The **live model sweep runs only in the harness, not the pytest gate** (a real constraint, recorded honestly): the agents are module-level singletons whose genai async client binds to the first event loop, and the suite's per-test synchronous `Runner.run` each spins a fresh loop — so piling more live model calls into the gate trips ADK's "Event loop is closed" teardown race (the §13.4 hazard, here on the Runner's own loop) and randomly fails a live test. The harness drives its whole sweep on **one** shared event loop (`asyncio.run` over `run_async`) and is robust; the pytest gate therefore carries the deterministic, key-free security core (always-green on any clone, with or without a key) and the harness is the live evidence a judge runs by hand. An early version that put a 7-call live sweep in the gate reliably broke a *different* random live test each run — empirically confirming the constraint.
- Eval-property correction (found while building): the adjudicator's `note` is internal grading rationale and naturally references the expected word when explaining a miss ("expected 'sun', got X") — so a "no expected-word in the note" assertion flags normal behavior, not an injection-induced leak. The genuine answer-leak surface is the **coach hint** (shown to actively help mid-attempt), which is guarded (`reveals_word=False`). The grader's real property is "an injection earns no success." Tightened the eval accordingly.
- Alternatives considered: an **agents-cli eval-set** — rejected (ill fit for property assertions; key-dependent; worse clean-clone story); **rejecting** malformed recall rather than sanitizing — rejected as learner-hostile (a real answer could be dropped) and unnecessary given the output clamp is the true containment; **adding injection-detection heuristics** that block recalls matching injection patterns — rejected as brittle and beside the point (the structural clamp neutralizes injections regardless of how they're phrased); **persisting the recall for richer analytics** — rejected, it would import PII/injection text into the store and break the minimal-PII guarantee; **a full live-sweep in CI** — rejected per the event-loop constraint above.
- Concept served: Security (the fourth banked concept) — STRIDE threat model, injection containment, recall-input validation, minimal-PII; also a third authored Agent Skill.
- Shows in: code, video (DoD #5 — run the eval, see it pass; the STRIDE skill on screen), writeup ("Provenance & safety posture").

## 2026-06-24 — adjudicate + scaffold: semantic grade + crutch tag validated against visible-cue ground truth
- Decision: implement §13.6 by making Graph B live, mirroring the §13.4/§13.5 hybrid one more layer down. `adjudicate` is a no-tools Gemini node that grades the recall **semantically** (hit / near_miss / variant / miss — never a string compare) and **proposes** which crutch the recall leaned on; a pure `_validate_adjudication(...)` keeps that tag only if the cue was actually still **visible** for that word and the recall succeeded (a miss, or a cue that wasn't shown, leans on nothing). The "which cues were visible" facts are computed deterministically by `present_masked_line` from the structural map + the session's masks (`_visible_cues`/`_available_cues`). `scaffold` is a hybrid no-tools Gemini node: the rhyme cue (level 1) and first letter (level 2) are deterministic facts handed to it; it picks the minimum-effective level for where the learner stalled and authors a meaning gloss (level 3), with `_validate_hint` clamping the level and forbidding regression. `present_masked_line` loads the persisted Course and renders the masked **stanza**; the graded attempt rides through `advance`/`scaffold` into the unchanged `memory_update`, closing the §13.5 loop (real tags now feed `profile_from_attempts`).
- Why: §2/§4 require the grade to be semantic and the crutch-dependence tag to be real reasoning ("got 'X' only because rhyme partner 'Y' was visible") — a regex can't do either. §8 requires the model output to be a validated proposal, never trusted raw; the visible-cue ground truth is the deterministic constraint that makes the tag honest and unit-testable without a key. Rehearsed live end-to-end: stanza 0 masks the slant-rhyme word *Immortality* with its partner *me* still visible; the live grader returned `hit` + `crutch_dependence=rhyme_partner` ("rhymes with the visible partner 'me'"), it persisted, and the profile's dominant cue became `rhyme_partner` — exactly what §13.5 strips sooner next run.
- Present the **stanza, not the single line**: the rhyme-partner crutch lives across lines, so rendering only the target line hid the partner from the grader. An early rehearsal on "We passed the Setting ___" had the model (correctly, given what it could see) credit the *"setting sun"* collocation and tag `none`; once the whole stanza is shown, the visible partner is assessable and the tag lands. This was a genuine correctness fix, not just cosmetics.
- Graph B is kept **MCP-free**: the rhyme/metrical/stopword facts are already in `analyze_poem`'s structural map, so `adjudicate`/`scaffold` need no tool call. This avoids reusing one stdio MCP session across event loops (the §13.4 "Event loop is closed" hazard) and preserves the single-MCP-live-test invariant. The two graphs connect through a small JSON **course store** (`save_course`/`load_course` beside `LearnerMemory`, keyed by `(poem_id, learner_id)`) — the blueprint §4 "Course ──► Session/Memory store" edge — so Graph B presents exactly what Graph A planned.
- Runnable end-to-end via the **agents-cli local playground** + ADK `Runner`/`RequestInput` (§9 item 9: the playground "is sufficient"); a committed custom front-end stays the §10 opt-in stretch. Verification was an uncommitted scratch rehearsal driving the real node coroutines with the live model, plus key-free unit tests for the deterministic halves (`_visible_cues`, `_validate_adjudication`, `_candidate_hints`, `_validate_hint`, the course store, outcome→route) and key-gated wiring/live tests.
- Alternatives considered: a **deterministic/string-compare grader** — rejected as the §2 "for-loop" failure mode for the one place semantic judgment is the point; letting the **LLM emit the crutch tag unconstrained** — rejected per §8 (it would over-claim cues that weren't shown, e.g. the collocation case above); **fully deterministic scaffold** — rejected because §4 lists `scaffold` as an LLM node whose job (the minimum hint for *where the learner stalled*) needs judgment, though the rhyme/first-letter facts stay deterministic; **calling the Prosody MCP from Graph B** for rhyme facts — rejected (the facts are already in the structural map, and a second live MCP session reintroduces the cross-event-loop hazard); **rebuilding the Course in Graph B** instead of persisting it — rejected as duplicating `curriculum_plan`'s LLM anchors/overlay; persisting it is the §4 seam and lossless.
- Concept served: Agent / multi-agent (ADK) — the `adjudicate ⇄ memory ⇄ curriculum_plan` interdependence that closes the adaptive loop; Security (minimal-PII store, fail-safe validation).

## 2026-06-24 — Learner Memory + adaptive overlay: LLM *chooses* the crutch, policy *applies* it
- Decision: implement §13.5 (the adaptation "money shot") by extending the §13.4 hybrid one layer up. A new `app/curriculum/memory.py` provides (a) `LearnerMemory`, a small JSON append log of recall `Attempt`s, and (b) `profile_from_attempts(...)`, a **deterministic** reduction to a per-poem crutch-dependence profile (`relied_on` = correct recalls that leaned on a cue; `missed_at` = failures at that cue's words; `dominant` = the cues ranked by lean). `curriculum_plan` then, **only when the learner has a lean signal**, calls a focused no-tools Gemini node (`_adaptive_planner`) that **chooses** which crutch class to strip next and writes a one-line diagnosis. That choice is a *proposal*: `_validate_directive(...)` rejects any class the poem doesn't actually exhibit, and the deterministic policy applies a valid directive by pulling that class's masks **one rung earlier** (`plan_course(..., adaptation=...)`). The existing §13.4 rationale agent then narrates the personalized schedule. With no history, `plan_course` is byte-for-byte its §13.4 self (cold start unchanged).
- Why: §4 step 4 attributes the verbs "infers / diagnoses / **chooses**" to the Architect, and §2 lists "adaptive crutch-removal" as something a `for` loop *cannot* do — so the *decision* of which cue to strip is an LLM act, not a `sort()`; a deterministic ranking here would invite exactly the "replace the agent with a loop" dismissal §2 warns against. But §8 ("LLM output is a proposal validated, never trusted raw") and the §13.4 settled hybrid require the *schedule* to stay deterministic and key-free-testable. The split — deterministic profile (the evidence) → LLM directive (the choice) → deterministic apply (the masks) → LLM rationale (the visible prose) — is the only reading that satisfies both. Rehearsed end-to-end: seeding a metrical-dependence history pulls the anchor words from session 2 (rung 3) into session 1 (rung 2) and the rationale shifts to *"Because you have developed a strong reliance on the poem's metrical cadence… we must strip this cue now."*
- State store = JSON, not SQLite (blueprint line 276 left this open, "whichever is simpler to demo"). JSON is human-readable (you can open `learner_memory.json` on camera for the money shot), needs no schema/migration, and matches the pervasive `to_dict`/`from_dict` dataclass convention. The `.gitignore` already keeps learner runtime state untracked (2026-06-22 entry); the store defaults under the already-ignored `var/` with a `BY_HEART_STATE_DIR` override, so no `.gitignore` change was needed. Minimal-PII (§8): the only identity stored is an opaque `learner_id`.
- Seam-first ordering (honest scope): the crutch-dependence **tag** that makes `relied_on` meaningful is emitted by `adjudicate`, which is **§13.6, not yet built**. §13.5 therefore builds and tests the *entire* adaptation machinery and demonstrates it on seeded history (deterministic, key-free); `memory_update` already persists well-formed attempts (and ignores the current stub payload so it records no noise), so §13.6 only has to emit real grades + tags to close the live loop. This is the blueprint's own build order (memory before adjudicate).
- Alternatives considered: a **fully deterministic re-plan** (rank the profile, strip the top cue earlier; LLM only narrates) — rejected as the §2 "for-loop" failure mode for the one place the blueprint most wants visible agent reasoning; an **LLM that emits the whole adapted schedule** — rejected for the same §8 reason §13.4 rejected it (drift on safety-critical mask positions); **SQLite** for the store — rejected as more ceremony than an append-only attempt log needs and worse to demo; threading the **raw `CrutchProfile`** into `plan_course` instead of a validated directive — rejected so the pure policy stays independent of both the store and the model (it takes only a typed directive it can validate-by-construction).
- Concept served: Agent / multi-agent (ADK) — the adaptive interdependence (planner ⇄ learner memory ⇄ rationale); Security (minimal-PII store, secrets-clean).
- Shows in: code, video (the personalized Deletion Rationale + the visible `learner_memory.json`), writeup.

## 2026-06-24 — curriculum_plan: deterministic crutch-removal ladder + LLM-authored rationale
- Decision: implement §13.4 as a **hybrid**, not a single LLM call. A pure, key-free module (`app/curriculum/policy.py`) computes the multi-session masking *schedule* — the escalating crutch-removal ladder (rung 1: mask the later word of a rhyme pair, partner still visible → rung 2: mask both members → rung 3: mask the anchor content words → rung 4: near-whole-line) — and the `curriculum_plan` node then calls Gemini (via `ctx.run_node`, no MCP) **only** to author the per-session **Deletion Rationale** prose. The schedule the Recall graph will consume is therefore deterministic and unit-testable; the LLM owns the visible reasoning artifact.
- Why: the blueprint pulls in two directions and the hybrid is the only reading that satisfies both. §4 lists `curriculum_plan` as an LLM node whose Deletion Rationale is the judge-facing "money shot"; §8/§11 demand the policy be "concrete, ordered, falsifiable" and verifiable without a key. Making the ladder deterministic keeps the masking positions reliable for Graph B and gradeable in CI; confining the LLM to the rationale keeps the agent doing reasoning a `for` loop cannot (naming *which* crutch each deletion strips and why) without letting it hallucinate the schedule.
- The load-bearing LLM dependency (§4 "curriculum_plan depends on prosody_analysis"): the **anchor words** are the genuine hand-off. `prosody_analysis` now commits a structured `anchor_words` judgment (ADK `output_schema` + `output_key`, which 2.3.0 confirms coexist with tools — the agent still calls the MCP), and those anchors drive the rung-3 deletions. The deterministic *phonetic* facts (rhyme-partner map, stress) are **not** laundered through the LLM (§8 "validated against the dict, never trusted raw"): `curriculum_plan` re-derives them with `analyze_poem()` on the admitted poem text (stashed in session state by the gate). A deterministic anchor heuristic is the fallback when no key/LLM is present, so the schedule is always well-formed.
- `history` (learner error patterns) is threaded through `plan_course(...)` as a typed but inert **seam** for §13.5's adaptive overlay — the place the "remove the crutch you actually leaned on" re-planning will attach — so adding it later doesn't reshape the signature.
- Alternatives considered: a **pure LlmAgent + a curriculum FunctionTool** (mirroring how `prosody_analysis` is an LlmAgent + MCP toolset) — rejected because the masking positions would then be only as reliable as the model's tool-discipline (it could skip the tool, pass bad args, or paraphrase the result), failing §11's gradeable-artifact bar; a **fully deterministic node with templated rationale** (rejected — no surfaced LLM reasoning, the exact "for-loop" failure mode §1 warns against; the templated strings survive only as the no-key fallback); having `prosody_analysis` emit the **whole structural map** via `output_schema` and consuming it directly (rejected — risks LLM drift/lossiness on the safety-critical rhyme/stress facts).
- Test-isolation note: the Build Pipeline shares one module-level stdio MCP toolset, and a stdio session cached on one `runner.run` event loop cannot be reused on the next ("Event loop is closed"). So exactly one live test exercises the MCP path (`prosody_analysis`); the curriculum live test drives the rationale agent (no MCP) over the real deterministic schedule. The full provenance→prosody(MCP)→curriculum pipeline was verified manually end-to-end (Course emitted with rungs 1–4 and a crutch-naming rationale per session).
- Concept served: Agent / multi-agent (ADK).
- Shows in: code, video (the Deletion Rationale is the on-screen reasoning artifact), writeup.

## 2026-06-24 — Prosody g2p fallback: gruut (offline), not g2p_en
- Decision: the Prosody MCP resolver is CMU-primary (`pronouncing`/`cmudict`, dict bundled in the wheel) → a deterministic poetic-elision normalizer (o'er→over, anchor'd→anchored, a-crowding→crowding) + morphological suffix backoff → **`gruut` (+`gruut-lang-en`)** as the grapheme-to-phoneme fallback for genuinely out-of-dictionary tokens. gruut's IPA output is mapped to ARPAbet so it's consistent with CMU for rhyme comparison.
- Why: measured the real gap — of 224 unique words across the 3 seed poems, only 6 miss CMU, and 5 of those are elisions/morphology the normalizer handles; the true predictor is needed only for rare words like `cornice` and (as the corpus grows) proper nouns like Millay's "Recuerdo". The blueprint (§8/§10) requires a real g2p library AND "setup verified on a clean clone" (§11/§12). `g2p_en` fails that gate: it downloads nltk data at runtime to `~/nltk_data` (not offline, writes outside the project), pulls a heavy numpy/nltk tree, prints download progress to stdout (breaks MCP stdout hygiene), and is currently broken against nltk 3.9.4 (wants `averaged_perceptron_tagger_eng`). `gruut` ships its English data in the `gruut-lang-en` wheel — verified it phonemizes `cornice` and `Recuerdo` **offline**, with zero runtime downloads and nothing committed to the repo (deps live in the gitignored `.venv`; repo content stays ~360 KB).
- Cost accepted: gruut adds ~75 MB to `.venv` (gitignored) and a compiled dep (`python-crfsuite`, prebuilt wheels exist) — disproportionate to today's gap, but it is the only clean-clone-offline option that truly predicts novel words/proper nouns, which the seed corpus will need.
- Alternatives considered: `g2p_en` (best-known, but the runtime-download/offline failures above); `eng_to_ipa` / dict-only libs (pure-Python but don't *predict* OOV — they flag unknowns, so "Recuerdo" wouldn't resolve, failing §8's "every word resolves"); a hand-rolled rule-based g2p (dependency-free, but weaker on proper nouns and less defensible as "a g2p library").
- Concept served: MCP server.

## 2026-06-24 — Corpus text canonicalization: body-only, and which edition the hash pins
- Decision: store each poem as **body-only** (no title line) UTF-8 with a single trailing newline and single blank lines between stanzas; the manifest `sha256` then binds those exact bytes. Where a poem has competing editions, the committed text matches the **public-domain source we recorded in `source_url`**, not the modern critical edition. Concretely: Dickinson is the regularized **1890 "The Chariot"** text from Gutenberg #12242 — five stanzas, omitting the "Or rather, he passed us" stanza and using "children played / Their lessons scarcely done" — *not* the 6-stanza Franklin/Johnson text. Frost and Whitman are likewise the bodies of their recorded sources (Whitman = deathbed-edition).
- Why: body-only keeps the unit of masking (Graph B) equal to a line of verse, with no title to special-case. Pinning the hash to the *recorded source's* edition keeps `source_url` honest — the cited page actually contains these bytes — and makes the corpus reproducible. The editorial-variant note matters: someone diffing our Dickinson against a modern anthology will see "missing stanza" and "wrong stanza 3," and that is correct-by-design, not corruption.
- Process note: each pasted text was proofed line-by-line before hashing; this caught a missing "a" in the Frost title (moot once title-dropped) and a stray double space in Dickinson's "centuries; but each". Small edits change the hash, so they were settled before the manifest recorded `sha256`.
- Alternatives considered: include the title line (rejected — adds a non-verse line the masking logic must exempt); use the modern critical edition for Dickinson (rejected — it doesn't match the 1890 `source_url`, breaking the provenance chain the gate asserts); normalize/reformat pasted text (rejected — store exactly what the source prints, so the hash means "this edition," not "our reflow").
- Concept served: Security / Agent skills.
- Shows in: code, writeup.

## 2026-06-24 — Provenance gate: sha256 content integrity + pure, testable policy
