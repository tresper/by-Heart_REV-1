## 2026-06-24 ŌĆö Security (┬¦13.7): STRIDE skill + injection/PII eval + recall-input validation
- Decision: bank the Security concept by hardening the one untrusted entry point, proving the guarantees with a runnable eval, and authoring a STRIDE `SKILL.md`. (a) **Recall-input validation** lives in a small pure module `app/security/recall_input.py` (`sanitize_recall`) applied at the single choke point `_recall_text` ŌĆö it bounds length, strips Unicode control/format codepoints (zero-width, bidi, BOM, soft hyphen) and collapses newlines, while preserving legitimate poetic tokens (`o'er`, `caf├®`, hyphens). One change covers both the adjudicator read and the stored `last_recall` the coach reuses. (b) **Prompt hardening**: both LLM instructions now state the recall is untrusted DATA, never an instruction (the coach also: never reveal the word). (c) The **structural containment was already present** ŌĆö `_validate_adjudication` clamps `outcome` to the legal vocabulary and `crutch_dependence` to the cues that were actually visible, and `_validate_hint` clamps the hint level ŌĆö so this entry promotes the ┬¦8 validated-proposal pattern to a stated security property: a fully-injection-swayed model still cannot emit an out-of-vocabulary grade, fabricate a crutch tag, or over-escalate a hint. (d) **Minimal-PII is structural** ŌĆö `Attempt` has no free-text recall field, so a recall's PII/injection payload is never persisted; no code change, the eval asserts it. (e) A new `.claude/skills/stride-threat-model/SKILL.md` maps each STRIDE category to By Heart's concrete control and its code location (a third authored Skill).
- Why: ┬¦5/┬¦8 require Security as one of the four banked concepts, anchored in the MVP. The system's attack surface is small by design (no accounts, no open-ended ingestion, one untrusted input), so the work is to make the existing controls explicit, hardened, and provable rather than to add machinery. DoD ┬¦11 #5 ("run the injection/PII eval and see it pass") is the gradeable artifact. Verified: the eval's 28 checks pass and the standalone harness exits 0 with the live model resisting every injection scenario (each injected recall graded a non-success: `miss`/`near_miss`).
- Eval shipped as a **runnable `evals/` harness + a deterministic pytest core**, NOT an agents-cli eval-set. ┬¦8/┬¦10 name agents-cli for the lifecycle, but DoD #5 only requires the eval *runs and passes on a clean clone*, and the settled project convention (┬¦13.5/┬¦13.6) is a key-free deterministic core plus a key-gated live layer. Security *properties* (injection resistance, PII non-leakage) are property assertions, not reference-answer matches, so agents-cli's eval-set format is an ill fit and key-dependent. `evals/scenarios.py` holds the synthetic injection + PII corpus as readable data; `evals/injection_pii_eval.py` runs it as a one-command PASS/FAIL report (`uv run python -m evals.injection_pii_eval`) and is imported by `tests/eval/` so `uv run pytest` stays the single gate.
- The **live model sweep runs only in the harness, not the pytest gate** (a real constraint, recorded honestly): the agents are module-level singletons whose genai async client binds to the first event loop, and the suite's per-test synchronous `Runner.run` each spins a fresh loop ŌĆö so piling more live model calls into the gate trips ADK's "Event loop is closed" teardown race (the ┬¦13.4 hazard, here on the Runner's own loop) and randomly fails a live test. The harness drives its whole sweep on **one** shared event loop (`asyncio.run` over `run_async`) and is robust; the pytest gate therefore carries the deterministic, key-free security core (always-green on any clone, with or without a key) and the harness is the live evidence a judge runs by hand. An early version that put a 7-call live sweep in the gate reliably broke a *different* random live test each run ŌĆö empirically confirming the constraint.
- Eval-property correction (found while building): the adjudicator's `note` is internal grading rationale and naturally references the expected word when explaining a miss ("expected 'sun', got X") ŌĆö so a "no expected-word in the note" assertion flags normal behavior, not an injection-induced leak. The genuine answer-leak surface is the **coach hint** (shown to actively help mid-attempt), which is guarded (`reveals_word=False`). The grader's real property is "an injection earns no success." Tightened the eval accordingly.
- Alternatives considered: an **agents-cli eval-set** ŌĆö rejected (ill fit for property assertions; key-dependent; worse clean-clone story); **rejecting** malformed recall rather than sanitizing ŌĆö rejected as learner-hostile (a real answer could be dropped) and unnecessary given the output clamp is the true containment; **adding injection-detection heuristics** that block recalls matching injection patterns ŌĆö rejected as brittle and beside the point (the structural clamp neutralizes injections regardless of how they're phrased); **persisting the recall for richer analytics** ŌĆö rejected, it would import PII/injection text into the store and break the minimal-PII guarantee; **a full live-sweep in CI** ŌĆö rejected per the event-loop constraint above.
- Concept served: Security (the fourth banked concept) ŌĆö STRIDE threat model, injection containment, recall-input validation, minimal-PII; also a third authored Agent Skill.
- Shows in: code, video (DoD #5 ŌĆö run the eval, see it pass; the STRIDE skill on screen), writeup ("Provenance & safety posture").

## 2026-06-24 ŌĆö adjudicate + scaffold: semantic grade + crutch tag validated against visible-cue ground truth
- Decision: implement ┬¦13.6 by making Graph B live, mirroring the ┬¦13.4/┬¦13.5 hybrid one more layer down. `adjudicate` is a no-tools Gemini node that grades the recall **semantically** (hit / near_miss / variant / miss ŌĆö never a string compare) and **proposes** which crutch the recall leaned on; a pure `_validate_adjudication(...)` keeps that tag only if the cue was actually still **visible** for that word and the recall succeeded (a miss, or a cue that wasn't shown, leans on nothing). The "which cues were visible" facts are computed deterministically by `present_masked_line` from the structural map + the session's masks (`_visible_cues`/`_available_cues`). `scaffold` is a hybrid no-tools Gemini node: the rhyme cue (level 1) and first letter (level 2) are deterministic facts handed to it; it picks the minimum-effective level for where the learner stalled and authors a meaning gloss (level 3), with `_validate_hint` clamping the level and forbidding regression. `present_masked_line` loads the persisted Course and renders the masked **stanza**; the graded attempt rides through `advance`/`scaffold` into the unchanged `memory_update`, closing the ┬¦13.5 loop (real tags now feed `profile_from_attempts`).
- Why: ┬¦2/┬¦4 require the grade to be semantic and the crutch-dependence tag to be real reasoning ("got 'X' only because rhyme partner 'Y' was visible") ŌĆö a regex can't do either. ┬¦8 requires the model output to be a validated proposal, never trusted raw; the visible-cue ground truth is the deterministic constraint that makes the tag honest and unit-testable without a key. Rehearsed live end-to-end: stanza 0 masks the slant-rhyme word *Immortality* with its partner *me* still visible; the live grader returned `hit` + `crutch_dependence=rhyme_partner` ("rhymes with the visible partner 'me'"), it persisted, and the profile's dominant cue became `rhyme_partner` ŌĆö exactly what ┬¦13.5 strips sooner next run.
- Present the **stanza, not the single line**: the rhyme-partner crutch lives across lines, so rendering only the target line hid the partner from the grader. An early rehearsal on "We passed the Setting ___" had the model (correctly, given what it could see) credit the *"setting sun"* collocation and tag `none`; once the whole stanza is shown, the visible partner is assessable and the tag lands. This was a genuine correctness fix, not just cosmetics.
- Graph B is kept **MCP-free**: the rhyme/metrical/stopword facts are already in `analyze_poem`'s structural map, so `adjudicate`/`scaffold` need no tool call. This avoids reusing one stdio MCP session across event loops (the ┬¦13.4 "Event loop is closed" hazard) and preserves the single-MCP-live-test invariant. The two graphs connect through a small JSON **course store** (`save_course`/`load_course` beside `LearnerMemory`, keyed by `(poem_id, learner_id)`) ŌĆö the blueprint ┬¦4 "Course ŌöĆŌöĆŌ¢║ Session/Memory store" edge ŌĆö so Graph B presents exactly what Graph A planned.
- Runnable end-to-end via the **agents-cli local playground** + ADK `Runner`/`RequestInput` (┬¦9 item 9: the playground "is sufficient"); a committed custom front-end stays the ┬¦10 opt-in stretch. Verification was an uncommitted scratch rehearsal driving the real node coroutines with the live model, plus key-free unit tests for the deterministic halves (`_visible_cues`, `_validate_adjudication`, `_candidate_hints`, `_validate_hint`, the course store, outcomeŌåÆroute) and key-gated wiring/live tests.
- Alternatives considered: a **deterministic/string-compare grader** ŌĆö rejected as the ┬¦2 "for-loop" failure mode for the one place semantic judgment is the point; letting the **LLM emit the crutch tag unconstrained** ŌĆö rejected per ┬¦8 (it would over-claim cues that weren't shown, e.g. the collocation case above); **fully deterministic scaffold** ŌĆö rejected because ┬¦4 lists `scaffold` as an LLM node whose job (the minimum hint for *where the learner stalled*) needs judgment, though the rhyme/first-letter facts stay deterministic; **calling the Prosody MCP from Graph B** for rhyme facts ŌĆö rejected (the facts are already in the structural map, and a second live MCP session reintroduces the cross-event-loop hazard); **rebuilding the Course in Graph B** instead of persisting it ŌĆö rejected as duplicating `curriculum_plan`'s LLM anchors/overlay; persisting it is the ┬¦4 seam and lossless.
- Concept served: Agent / multi-agent (ADK) ŌĆö the `adjudicate Ōćä memory Ōćä curriculum_plan` interdependence that closes the adaptive loop; Security (minimal-PII store, fail-safe validation).

## 2026-06-24 ŌĆö Learner Memory + adaptive overlay: LLM *chooses* the crutch, policy *applies* it
- Decision: implement ┬¦13.5 (the adaptation "money shot") by extending the ┬¦13.4 hybrid one layer up. A new `app/curriculum/memory.py` provides (a) `LearnerMemory`, a small JSON append log of recall `Attempt`s, and (b) `profile_from_attempts(...)`, a **deterministic** reduction to a per-poem crutch-dependence profile (`relied_on` = correct recalls that leaned on a cue; `missed_at` = failures at that cue's words; `dominant` = the cues ranked by lean). `curriculum_plan` then, **only when the learner has a lean signal**, calls a focused no-tools Gemini node (`_adaptive_planner`) that **chooses** which crutch class to strip next and writes a one-line diagnosis. That choice is a *proposal*: `_validate_directive(...)` rejects any class the poem doesn't actually exhibit, and the deterministic policy applies a valid directive by pulling that class's masks **one rung earlier** (`plan_course(..., adaptation=...)`). The existing ┬¦13.4 rationale agent then narrates the personalized schedule. With no history, `plan_course` is byte-for-byte its ┬¦13.4 self (cold start unchanged).
- Why: ┬¦4 step 4 attributes the verbs "infers / diagnoses / **chooses**" to the Architect, and ┬¦2 lists "adaptive crutch-removal" as something a `for` loop *cannot* do ŌĆö so the *decision* of which cue to strip is an LLM act, not a `sort()`; a deterministic ranking here would invite exactly the "replace the agent with a loop" dismissal ┬¦2 warns against. But ┬¦8 ("LLM output is a proposal validated, never trusted raw") and the ┬¦13.4 settled hybrid require the *schedule* to stay deterministic and key-free-testable. The split ŌĆö deterministic profile (the evidence) ŌåÆ LLM directive (the choice) ŌåÆ deterministic apply (the masks) ŌåÆ LLM rationale (the visible prose) ŌĆö is the only reading that satisfies both. Rehearsed end-to-end: seeding a metrical-dependence history pulls the anchor words from session 2 (rung 3) into session 1 (rung 2) and the rationale shifts to *"Because you have developed a strong reliance on the poem's metrical cadenceŌĆ” we must strip this cue now."*
- State store = JSON, not SQLite (blueprint line 276 left this open, "whichever is simpler to demo"). JSON is human-readable (you can open `learner_memory.json` on camera for the money shot), needs no schema/migration, and matches the pervasive `to_dict`/`from_dict` dataclass convention. The `.gitignore` already keeps learner runtime state untracked (2026-06-22 entry); the store defaults under the already-ignored `var/` with a `BY_HEART_STATE_DIR` override, so no `.gitignore` change was needed. Minimal-PII (┬¦8): the only identity stored is an opaque `learner_id`.
- Seam-first ordering (honest scope): the crutch-dependence **tag** that makes `relied_on` meaningful is emitted by `adjudicate`, which is **┬¦13.6, not yet built**. ┬¦13.5 therefore builds and tests the *entire* adaptation machinery and demonstrates it on seeded history (deterministic, key-free); `memory_update` already persists well-formed attempts (and ignores the current stub payload so it records no noise), so ┬¦13.6 only has to emit real grades + tags to close the live loop. This is the blueprint's own build order (memory before adjudicate).
- Alternatives considered: a **fully deterministic re-plan** (rank the profile, strip the top cue earlier; LLM only narrates) ŌĆö rejected as the ┬¦2 "for-loop" failure mode for the one place the blueprint most wants visible agent reasoning; an **LLM that emits the whole adapted schedule** ŌĆö rejected for the same ┬¦8 reason ┬¦13.4 rejected it (drift on safety-critical mask positions); **SQLite** for the store ŌĆö rejected as more ceremony than an append-only attempt log needs and worse to demo; threading the **raw `CrutchProfile`** into `plan_course` instead of a validated directive ŌĆö rejected so the pure policy stays independent of both the store and the model (it takes only a typed directive it can validate-by-construction).
- Concept served: Agent / multi-agent (ADK) ŌĆö the adaptive interdependence (planner Ōćä learner memory Ōćä rationale); Security (minimal-PII store, secrets-clean).
- Shows in: code, video (the personalized Deletion Rationale + the visible `learner_memory.json`), writeup.

## 2026-06-24 ŌĆö curriculum_plan: deterministic crutch-removal ladder + LLM-authored rationale
- Decision: implement ┬¦13.4 as a **hybrid**, not a single LLM call. A pure, key-free module (`app/curriculum/policy.py`) computes the multi-session masking *schedule* ŌĆö the escalating crutch-removal ladder (rung 1: mask the later word of a rhyme pair, partner still visible ŌåÆ rung 2: mask both members ŌåÆ rung 3: mask the anchor content words ŌåÆ rung 4: near-whole-line) ŌĆö and the `curriculum_plan` node then calls Gemini (via `ctx.run_node`, no MCP) **only** to author the per-session **Deletion Rationale** prose. The schedule the Recall graph will consume is therefore deterministic and unit-testable; the LLM owns the visible reasoning artifact.
- Why: the blueprint pulls in two directions and the hybrid is the only reading that satisfies both. ┬¦4 lists `curriculum_plan` as an LLM node whose Deletion Rationale is the judge-facing "money shot"; ┬¦8/┬¦11 demand the policy be "concrete, ordered, falsifiable" and verifiable without a key. Making the ladder deterministic keeps the masking positions reliable for Graph B and gradeable in CI; confining the LLM to the rationale keeps the agent doing reasoning a `for` loop cannot (naming *which* crutch each deletion strips and why) without letting it hallucinate the schedule.
- The load-bearing LLM dependency (┬¦4 "curriculum_plan depends on prosody_analysis"): the **anchor words** are the genuine hand-off. `prosody_analysis` now commits a structured `anchor_words` judgment (ADK `output_schema` + `output_key`, which 2.3.0 confirms coexist with tools ŌĆö the agent still calls the MCP), and those anchors drive the rung-3 deletions. The deterministic *phonetic* facts (rhyme-partner map, stress) are **not** laundered through the LLM (┬¦8 "validated against the dict, never trusted raw"): `curriculum_plan` re-derives them with `analyze_poem()` on the admitted poem text (stashed in session state by the gate). A deterministic anchor heuristic is the fallback when no key/LLM is present, so the schedule is always well-formed.
- `history` (learner error patterns) is threaded through `plan_course(...)` as a typed but inert **seam** for ┬¦13.5's adaptive overlay ŌĆö the place the "remove the crutch you actually leaned on" re-planning will attach ŌĆö so adding it later doesn't reshape the signature.
- Alternatives considered: a **pure LlmAgent + a curriculum FunctionTool** (mirroring how `prosody_analysis` is an LlmAgent + MCP toolset) ŌĆö rejected because the masking positions would then be only as reliable as the model's tool-discipline (it could skip the tool, pass bad args, or paraphrase the result), failing ┬¦11's gradeable-artifact bar; a **fully deterministic node with templated rationale** (rejected ŌĆö no surfaced LLM reasoning, the exact "for-loop" failure mode ┬¦1 warns against; the templated strings survive only as the no-key fallback); having `prosody_analysis` emit the **whole structural map** via `output_schema` and consuming it directly (rejected ŌĆö risks LLM drift/lossiness on the safety-critical rhyme/stress facts).
- Test-isolation note: the Build Pipeline shares one module-level stdio MCP toolset, and a stdio session cached on one `runner.run` event loop cannot be reused on the next ("Event loop is closed"). So exactly one live test exercises the MCP path (`prosody_analysis`); the curriculum live test drives the rationale agent (no MCP) over the real deterministic schedule. The full provenanceŌåÆprosody(MCP)ŌåÆcurriculum pipeline was verified manually end-to-end (Course emitted with rungs 1ŌĆō4 and a crutch-naming rationale per session).
- Concept served: Agent / multi-agent (ADK).
- Shows in: code, video (the Deletion Rationale is the on-screen reasoning artifact), writeup.

## 2026-06-24 ŌĆö Prosody g2p fallback: gruut (offline), not g2p_en
- Decision: the Prosody MCP resolver is CMU-primary (`pronouncing`/`cmudict`, dict bundled in the wheel) ŌåÆ a deterministic poetic-elision normalizer (o'erŌåÆover, anchor'dŌåÆanchored, a-crowdingŌåÆcrowding) + morphological suffix backoff ŌåÆ **`gruut` (+`gruut-lang-en`)** as the grapheme-to-phoneme fallback for genuinely out-of-dictionary tokens. gruut's IPA output is mapped to ARPAbet so it's consistent with CMU for rhyme comparison.
- Why: measured the real gap ŌĆö of 224 unique words across the 3 seed poems, only 6 miss CMU, and 5 of those are elisions/morphology the normalizer handles; the true predictor is needed only for rare words like `cornice` and (as the corpus grows) proper nouns like Millay's "Recuerdo". The blueprint (┬¦8/┬¦10) requires a real g2p library AND "setup verified on a clean clone" (┬¦11/┬¦12). `g2p_en` fails that gate: it downloads nltk data at runtime to `~/nltk_data` (not offline, writes outside the project), pulls a heavy numpy/nltk tree, prints download progress to stdout (breaks MCP stdout hygiene), and is currently broken against nltk 3.9.4 (wants `averaged_perceptron_tagger_eng`). `gruut` ships its English data in the `gruut-lang-en` wheel ŌĆö verified it phonemizes `cornice` and `Recuerdo` **offline**, with zero runtime downloads and nothing committed to the repo (deps live in the gitignored `.venv`; repo content stays ~360 KB).
- Cost accepted: gruut adds ~75 MB to `.venv` (gitignored) and a compiled dep (`python-crfsuite`, prebuilt wheels exist) ŌĆö disproportionate to today's gap, but it is the only clean-clone-offline option that truly predicts novel words/proper nouns, which the seed corpus will need.
- Alternatives considered: `g2p_en` (best-known, but the runtime-download/offline failures above); `eng_to_ipa` / dict-only libs (pure-Python but don't *predict* OOV ŌĆö they flag unknowns, so "Recuerdo" wouldn't resolve, failing ┬¦8's "every word resolves"); a hand-rolled rule-based g2p (dependency-free, but weaker on proper nouns and less defensible as "a g2p library").
- Concept served: MCP server.
- Shows in: code, writeup.

## 2026-06-24 ŌĆö Corpus text canonicalization: body-only, and which edition the hash pins
- Decision: store each poem as **body-only** (no title line) UTF-8 with a single trailing newline and single blank lines between stanzas; the manifest `sha256` then binds those exact bytes. Where a poem has competing editions, the committed text matches the **public-domain source we recorded in `source_url`**, not the modern critical edition. Concretely: Dickinson is the regularized **1890 "The Chariot"** text from Gutenberg #12242 ŌĆö five stanzas, omitting the "Or rather, he passed us" stanza and using "children played / Their lessons scarcely done" ŌĆö *not* the 6-stanza Franklin/Johnson text. Frost and Whitman are likewise the bodies of their recorded sources (Whitman = deathbed-edition).
- Why: body-only keeps the unit of masking (Graph B) equal to a line of verse, with no title to special-case. Pinning the hash to the *recorded source's* edition keeps `source_url` honest ŌĆö the cited page actually contains these bytes ŌĆö and makes the corpus reproducible. The editorial-variant note matters: someone diffing our Dickinson against a modern anthology will see "missing stanza" and "wrong stanza 3," and that is correct-by-design, not corruption.
- Process note: each pasted text was proofed line-by-line before hashing; this caught a missing "a" in the Frost title (moot once title-dropped) and a stray double space in Dickinson's "centuries; but each". Small edits change the hash, so they were settled before the manifest recorded `sha256`.
- Alternatives considered: include the title line (rejected ŌĆö adds a non-verse line the masking logic must exempt); use the modern critical edition for Dickinson (rejected ŌĆö it doesn't match the 1890 `source_url`, breaking the provenance chain the gate asserts); normalize/reformat pasted text (rejected ŌĆö store exactly what the source prints, so the hash means "this edition," not "our reflow").
- Concept served: Security / Agent skills.
- Shows in: code, writeup.

## 2026-06-24 ŌĆö Deletion theory: "removing crutches"
- Decision: the deletion policy strips the prosodic cue a learner is currently relying on (visible rhyme partner, metrical regularity, syntactic momentum), escalating by removing cues rather than masking a fixed ratio.
